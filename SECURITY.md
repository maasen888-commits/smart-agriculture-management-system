# Security Policy

感谢你帮助改进本项目的安全性。

## 支持范围

当前主要维护分支：

- `main`

当前公开版本仍在快速迭代中。我们会优先处理以下范围内的问题：

- 认证与权限绕过
- 密钥、令牌、敏感配置泄露
- 越权访问 API
- 任务调度导致的数据破坏或权限提升
- 文件上传与对象存储访问漏洞
- WebSocket、代理层、部署配置带来的安全风险

## 如何报告安全问题

请不要在公开 Issue 中直接披露漏洞细节、利用方式、真实密钥、客户资料或内部部署信息。

建议按以下顺序处理：

1. 如果仓库已启用 GitHub 私密安全报告，请优先使用 GitHub Security Advisory / Private Vulnerability Reporting。
2. 如果没有启用私密安全报告，请先提交一个最小化的公开 Issue，标题可写为 `Security Contact Needed`，只说明“需要私密沟通入口”，不要公开粘贴漏洞细节。
3. 也可以通过维护者在 GitHub 主页展示的私密联系方式单独联系。

## 报告时建议包含的信息

为了提高处理效率，建议提供：

1. 影响范围
2. 复现步骤
3. 实际结果与预期结果
4. 影响评估
5. 临时缓解建议
6. 如适用，附最小 PoC，但不要包含真实生产数据

## 响应原则

我们会尽量按以下节奏响应：

- 3 个工作日内确认收到
- 7 个工作日内给出初步判断
- 如问题成立，尽快安排修复与版本说明

这不是强 SLA，但会作为维护目标。

## 公开披露原则

在问题修复前，请不要公开披露以下内容：

- 完整利用链
- 可直接复用的攻击脚本
- 真实访问地址
- 真实密钥、系统 ID、客户数据或截图

如需发布修复说明，建议在补丁发布后再公开细节。

## 不在受理范围的内容

以下内容通常不视为安全漏洞：

- 仅开发环境示例配置中的占位值
- 没有实际权限提升或数据影响的普通报错
- 缺少业务上下文的“理论性”问题
- 第三方依赖的泛泛安全讨论，但没有结合本项目实际利用路径

## 额外说明

本仓库为公开源码、非商用许可项目。安全报告同样请避免泄露任何私有部署信息与合作方资料。
